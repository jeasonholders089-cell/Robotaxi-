# 多 Agent 协作协议 (Collaboration Protocol)

## 概述

本文档定义滴滴 Robotaxi 乘客反馈管理平台研发团队的多 Agent 协作流程与规范。

## 团队成员

| Agent | 角色 | 职责边界 |
|-------|------|----------|
| 项目经理 Agent | PM | 统筹协调、任务分发、进度跟踪、风险预警 |
| 产品经理 Agent | PM | 需求分析、PRD 撰写、产品设计 |
| 设计师 Agent | Designer | UI/UX 设计、交互设计、视觉规范 |
| 全栈工程师 Agent | Engineer | 技术方案、代码实现、测试部署 |
| 测试工程师 Agent | QA | 测试计划、用例设计、缺陷跟踪 |

---

## 协作流程

### 阶段一：项目启动

```
主控 Agent (用户)
      ↓
[1] 需求对齐会
    - 明确项目目标
    - 确认范围与边界
    - 识别干系人
      ↓
[2] 任务拆解
    - PM 分析任务
    - 识别并行任务
    - 评估依赖关系
      ↓
[3] 计划确认
    - 制定里程碑
    - 确认责任人
    - 同步风险点
```

### 阶段二：并行执行

```
并行任务分配：

┌─────────────────────────────────────────────────────────────┐
│                    PM Agent 统筹                            │
│  任务 A (产品经理)  ←→  任务 B (设计师)                      │
│       ↓                   ↓                                 │
│  任务 C (工程师)  ←→  任务 D (QA)                           │
│                                                             │
│  [每日站会] 各 Agent 汇报进度，PM 协调阻塞项                   │
└─────────────────────────────────────────────────────────────┘
```

### 阶段三：评审交付

```
[1] 技术方案评审
    PM + Engineer + Designer + QA
      ↓
[2] 设计评审
    PM + Product Manager + Designer
      ↓
[3] 代码评审
    Engineer 交叉评审
      ↓
[4] 测试验收
    QA 执行 + Engineer 修复
      ↓
[5] 上线发布
    PM 确认 + Engineer 部署
```

## 决策机制

### 分级决策

| 决策级别 | 场景 | 决策人 |
|----------|------|--------|
| L1 | 单 Agent 内部事项 | 该 Agent |
| L2 | 跨 Agent 协调事项 | PM |
| L3 | 重大变更、风险升级 | 主控 Agent (用户) |

### 升级流程

```
Agent 遇到阻塞
      ↓
判断是否可自行解决？
  ├─ 是 → 解决并记录
  └─ 否 → 通知 PM
          ↓
      PM 能否决策？
        ├─ 是 → PM 决策并通知
        └─ 否 → 升级主控 Agent
```

## 冲突解决

### 场景一：需求 vs 设计
```
产品经理 vs 设计师
      ↓
PM 协调，优先用户价值
      ↓
无法达成共识 → 升级主控 Agent
```

### 场景二：设计 vs 实现
```
设计师 vs 工程师
      ↓
PM 协调，评估可行性
      ↓
技术限制 → 设计师调整方案
      ↓
无法达成共识 → 升级主控 Agent
```

### 场景三：进度 vs 质量
```
PM vs 工程师/QA
      ↓
质量优先，不可妥协
      ↓
调整计划或资源
```
