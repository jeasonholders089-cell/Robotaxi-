# 项目开发规范 (Project Development Norm)

> **核心原则**: 文档先行，coding在后。思考清楚再动手，避免返工和浪费。

---

## 一、规范目标

1. **减少返工** - 通过前置文档评审，避免开发后推倒重来
2. **提升质量** - 明确标准再实现，减少迭代次数
3. **便于追溯** - 文档即决策记录，可追溯设计意图
4. **协作顺畅** - 各方对齐后再执行，减少沟通成本

---

## 二、开发流程 (Document-First Workflow)

```
┌─────────────────────────────────────────────────────────────────┐
│                     文档先行阶段                                  │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐│
│  │ 需求文档 │→│ 设计文档 │→│ 技术方案 │→│ 测试计划 │→│ 评审通过 ││
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘  └────┬────┘│
└─────────────────────────────────────────────────────────────┼───┘
                                                              │
                                                              ▼
                     代码实现阶段                                │
  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │ 代码编写 │→│ 单元测试 │→│ 代码评审 │→│ 测试验证 │→│ 合并部署 │
  └─────────┘  └─────────┘  └─────────┘  └─────────┘  └─────────┘
```

## 三、阶段门禁 (Stage Gates)

### 阶段 1：需求门禁

| 检查项 | 完成标准 | 责任人 |
|--------|----------|--------|
| PRD 文档 | 包含功能描述、用户场景、验收标准 | 产品经理 |
| 需求评审 | 相关方评审通过 | 项目经理 |
| 任务拆分 | 细化到可执行粒度 | 项目经理 |

**门禁条件**: PRD 文档存在 + 评审签字 ✓

---

### 阶段 2：设计门禁

| 检查项 | 完成标准 | 责任人 |
|--------|----------|--------|
| UI/UX 设计稿 | 高保真原型 + 交互说明 | 设计师 |
| 技术方案 | 架构设计 + 接口设计 + 数据模型 | 工程师 |
| 测试计划 | 用例设计 + 覆盖率目标 | 测试工程师 |

**门禁条件**: 设计稿评审通过 + 技术方案评审通过 ✓

---

### 阶段 3：开发门禁

| 检查项 | 完成标准 | 责任人 |
|--------|----------|--------|
| 核心代码 | 功能实现完成 | 工程师 |
| 单元测试 | 覆盖率 ≥ 80% | 工程师 |
| 代码自测 | 无 P0/P1 缺陷 | 工程师 |

**门禁条件**: 代码评审通过 + CI 绿灯 ✓

---

### 阶段 4：测试门禁

| 检查项 | 完成标准 | 责任人 |
|--------|----------|--------|
| 功能测试 | P0 用例 100% 通过 | 测试工程师 |
| 回归测试 | 核心流程无回归 | 测试工程师 |
| 缺陷修复 | P0/P1 缺陷全部修复 | 工程师 |

**门禁条件**: 测试报告评审通过 ✓

---

### 阶段 5：发布门禁

| 检查项 | 完成标准 | 责任人 |
|--------|----------|--------|
| UAT 验收 | 产品经理确认 | 产品经理 |
| 部署文档 | 更新完成 | 工程师 |
| 监控告警 | 验证可用 | 工程师 |

**门禁条件**: 所有门禁签字 ✓

## 四、文档先行约束 (Self-Discipline Rules)

### 强制约束 (Hard Constraints)

| 约束 | 描述 | 违规处理 |
|------|------|----------|
| **R-01** | 不写设计文档不允许写代码 | 代码不被接受 |
| **R-02** | 不通过评审不允许开发 | 返工处理 |
| **R-03** | 不写测试用例不允许提测 | 测试报告无效 |
| **R-04** | P0 缺陷未修复不允许发布 | 发布阻断 |

### 建议约束 (Soft Constraints)

| 约束 | 描述 | 建议处理 |
|------|------|----------|
| **S-01** | 文档应该先于代码 1-2 天完成 | 提前规划 |
| **S-02** | 复杂方案应先做原型验证 | 降低风险 |
| **S-03** | 设计变更应更新文档 | 保持同步 |
| **S-04** | 代码即文档，注释随代码 | 便于维护 |

## 五、文档清单 (Document Checklist)

### 需求阶段

- [ ] **PRD** - 产品需求文档
- [ ] **用户故事** - 功能点拆解
- [ ] **验收标准** - 明确可测量的标准
- [ ] **任务拆分** - WBS 工作分解结构

### 设计阶段

- [ ] **UI/UX 设计稿** - 原型 + 交互说明
- [ ] **技术方案** - 架构设计、接口设计、数据模型
- [ ] **测试计划** - 测试策略、用例设计
- [ ] **评审记录** - 评审意见和结论

### 开发阶段

- [ ] **代码** - 符合规范的实现
- [ ] **单元测试** - 测试代码
- [ ] **代码评审报告** - 评审意见
- [ ] **API 文档** - 接口说明

### 测试阶段

- [ ] **测试用例** - 执行的用例
- [ ] **测试报告** - 执行结果
- [ ] **缺陷报告** - 缺陷列表和状态
- [ ] **回归报告** - 回归测试结果

### 发布阶段

- [ ] **部署文档** - 部署步骤
- [ ] **变更记录** - 本次变更说明
- [ ] **上线检查清单** - 验证项
- [ ] **复盘报告** - 项目总结

## 六、违反规范的自我惩罚

| 违规场景 | 自我惩罚 |
|----------|----------|
| 未写文档直接写代码 | 停止开发，补全文档后重新评审 |
| 评审未通过擅自开发 | 返工，已写代码作废 |
| P0 缺陷未修复发布 | 回滚版本，重新走发布流程 |
| 文档与代码不一致 | 停止新功能开发，先同步文档 |

---

## 七、执行承诺

```
我 (执行者) 承诺：

1. ✅ 在编写代码之前，必定先完成相关文档
2. ✅ 在文档评审通过之前，不会开始编码
3. ✅ 在测试用例编写之前，不会提测
4. ✅ 在所有 P0 缺陷修复之前，不会申请发布
5. ✅ 任何设计变更，都会先更新文档

签名：________________
日期：________________
```

## 八、项目文档结构

### 目录规范

```
乘客反馈管理平台/
├── docs/
│   ├── 00_任务书/              # 任务来源
│   ├── 01_需求文档/             # PRD、用户故事、验收标准
│   ├── 02_设计文档/             # UI/UX、原型、组件规范
│   ├── 03_技术文档/             # 技术方案、数据库、API
│   ├── 04_测试文档/             # 测试计划、用例、报告
│   └── 05_项目报告/             # 部署、变更、复盘
├── src/                        # 源代码
├── tests/                      # 测试代码
└── README.md                   # 项目入口
```

### 文档命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 需求文档 | `PRD-{模块}.md` | `PRD-反馈列表.md` |
| 设计文档 | `{类型}-{模块}.md` | `UI-仪表盘.md` |
| 技术文档 | `技术方案-{模块}.md` | `技术方案-AI分析.md` |
| 测试文档 | `用例-{模块}.md` | `用例-反馈提交.md` |
| 评审记录 | `评审-{阶段}-{日期}.md` | `评审-需求-20260416.md` |

### 阶段产出要求

| 阶段 | 必须产出 | 评审要求 |
|------|----------|----------|
| 需求阶段 | PRD + 验收标准 | 评审签字 |
| 设计阶段 | UI稿 + 组件规范 | 评审签字 |
| 技术阶段 | 技术方案 + API文档 | 评审签字 |
| 测试阶段 | 测试计划 + 用例 | 评审签字 |
| 发布阶段 | 部署文档 + 复盘报告 | 评审签字 |
