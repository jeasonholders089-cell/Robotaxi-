# 多 Agent 协作与文档先行项目经验

**日期**: 2026-04-16
**项目**: Robotaxi 乘客反馈管理平台
**类型**: 经验总结

---

## 一、项目概述

使用 Claude Code 多 Agent 协作系统，从 0 到 1 构建了一个完整的乘客反馈管理平台产品，包含：
- 需求文档（PRD）
- UI/UX 设计文档
- 技术方案文档
- 测试计划文档
- 150 条模拟数据

## 二、多 Agent 协作经验

### 2.1 协作模式

**采用模式**: 主控 Agent（项目经理角色）+ 多个专业 Agent 并行执行

```
主控 Agent (我)
      ↓ 任务分发
┌─────────────────────────────────────┐
│  并行执行                           │
│  产品经理 Agent → PRD               │
│  设计师 Agent → UI/UX 设计         │
│  全栈工程师 Agent → 技术方案         │
│  测试工程师 Agent → 测试计划        │
└─────────────────────────────────────┘
      ↓ 结果汇总
    主控 Agent 评审
```

### 2.2 协作流程

| 阶段 | 动作 | 说明 |
|------|------|------|
| 1. 任务分发 | 主控 Agent 启动多个 Agent | 使用 `run_in_background: true` 并行执行 |
| 2. 独立执行 | 各 Agent 独立产出 | 每个 Agent 专注自己领域 |
| 3. 结果汇总 | 主控 Agent 收集产出 | TaskOutput 获取结果 |
| 4. 文档评审 | 主控 Agent 评审 | 发现问题，记录修改点 |
| 5. 修改执行 | 启动修复 Agent | 串行或并行修复 |
| 6. 最终确认 | 用户确认 | 通过后进入下一阶段 |

### 2.3 Agent 执行策略

| 策略 | 适用场景 | 说明 |
|------|----------|------|
| 并行执行 | 独立任务、无依赖 | 效率最大化 |
| 串行执行 | 有依赖关系 | 等待前置任务完成 |
| 重试机制 | 服务繁忙(529错误) | 等待后重新启动 |

### 2.4 经验教训

| 教训 | 问题描述 | 解决方案 |
|------|----------|----------|
| 服务限流 | 并行启动 4 个 Agent 遇到 529 错误 | 分批启动或重试 |
| 上下文丢失 | Agent 返回时可能丢失输出 | 验证输出文件存在 |
| 指令复杂度 | 一次性给太多任务 | 拆分为独立 Agent |

## 三、文档先行规范经验

### 3.1 规范核心

**原则**: 文档先行，coding 在后。思考清楚再动手。

**门禁流程**:
```
需求文档 → 设计文档 → 技术方案 → 测试计划 → 代码实现
    ↓           ↓           ↓           ↓           ↓
  评审通过    评审通过    评审通过    评审通过    测试通过
```

### 3.2 五阶段门禁

| 阶段 | 必须产出 | 评审要求 |
|------|----------|----------|
| 需求阶段 | PRD + 验收标准 | 评审签字 |
| 设计阶段 | UI稿 + 组件规范 | 评审签字 |
| 技术阶段 | 技术方案 + API文档 | 评审签字 |
| 测试阶段 | 测试计划 + 用例 | 评审签字 |
| 发布阶段 | 部署文档 + 复盘报告 | 评审签字 |

### 3.3 自我约束规则

| 规则 | 描述 |
|------|------|
| R-01 | 不写设计文档不允许写代码 |
| R-02 | 不通过评审不允许开发 |
| R-03 | 不写测试用例不允许提测 |
| R-04 | P0 缺陷未修复不允许发布 |

### 3.4 经验教训

| 教训 | 问题描述 | 解决方案 |
|------|----------|----------|
| 功能蔓延 | PRD 中包含了不需要的权限管理功能 | 与用户确认范围，不做假设 |
| 模块拆分 | 最初设计了独立的"导出中心"模块 | 功能应属于操作，而非独立模块 |
| 评审深度 | 第一轮评审未发现所有问题 | 需要多轮评审，用户参与确认 |

## 四、用户决策管理经验

### 4.1 需要明确确认的关键决策

| 决策项 | 问题 | 重要性 |
|--------|------|--------|
| 用户系统 | 是否需要登录认证？ | 🔴 高 |
| 权限管理 | 是否需要角色权限？ | 🔴 高 |
| 技术栈 | 前端/后端/数据库选型？ | 🔴 高 |
| 部署方式 | 部署到哪里？ | 🔴 高 |
| API 密钥 | AI 能力用谁的 API？ | 🔴 高 |
| 交付标准 | 什么叫"完成"？ | 🔴 高 |

### 4.2 决策记录方式

在项目配置文件中记录所有已确认的决策：

```markdown
## 已确认的决策

| 决策项 | 结论 | 说明 |
|--------|------|------|
| 用户系统 | 暂不接入 | 后期再考虑 |
| 权限管理 | 不需要 | 内部工具 |
```

### 4.3 经验教训

| 教训 | 问题描述 | 解决方案 |
|------|----------|----------|
| 假设用户意图 | 假设需要权限管理 | 用户明确说"不需要"，应立即删除 |
| 过度设计 | 设计了独立的导出中心 | 导出是操作，不是模块 |
| 范围蔓延 | 建议增加 Landing Page 等 | 先确认用户是否需要 |

## 五、文档结构经验

### 5.1 推荐的文档目录结构

```
项目/
├── docs/
│   ├── 00_任务书/              # 任务来源
│   ├── 01_需求文档/             # PRD、用户故事
│   ├── 02_设计文档/             # UI/UX、原型
│   ├── 03_技术文档/             # 技术方案、数据库
│   ├── 04_测试文档/             # 测试计划、用例
│   └── 05_项目报告/             # 部署、复盘
├── data/
│   └── mock/                   # 模拟数据
│       └── README.md          # 数据说明
└── README.md                  # 项目入口
```

### 5.2 文档命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 需求文档 | `PRD-{模块}.md` | `PRD-反馈列表.md` |
| 设计文档 | `UI-{模块}.md` | `UI-仪表盘.md` |
| 技术文档 | `技术方案-{模块}.md` | `技术方案-AI分析.md` |
| 测试文档 | `测试计划-{模块}.md` | `测试计划-反馈提交.md` |
| 经验文档 | `YYYY-MM-DD_主题.md` | `2026-04-16_xxx经验.md` |

### 5.3 文档同步机制

当某一处文档更新时，需要同步检查并更新相关文档：

```
模拟数据更新 → PRD 字段定义 → 技术方案数据库 → 测试数据说明
     ↓              ↓               ↓              ↓
   同步更新      同步更新       同步更新        同步更新
```

## 六、模拟数据经验

### 6.1 数据生成时机

| 时机 | 说明 |
|------|------|
| 文档先行之后 | 文档定义完成后再生成数据 |
| 开发之前 | 数据就绪后可立即用于开发测试 |

### 6.2 数据规格应与文档一致

| 来源 | 内容 |
|------|------|
| PRD 4.1 | 数据字段定义 |
| PRD 4.4 | 数据量、评分分布要求 |
| 模拟数据 | 实际生成的数据 |

**经验**: 三者必须保持一致，PRD 定义 → 技术方案实现 → 模拟数据验证。

### 6.3 字段扩展流程

```
提出新字段 → 评估价值 → 用户确认 → 更新 PRD → 更新技术方案 → 更新模拟数据
```

## 七、PRD 评审检查清单

### 7.1 功能完整性

- [ ] 任务书三大模块是否全部覆盖？
- [ ] 每个功能点是否有明确的验收标准？
- [ ] 是否有遗漏的功能细节？

### 7.2 用户决策一致性

- [ ] 是否包含不需要的权限管理？
- [ ] 是否有与用户决策矛盾的内容？
- [ ] 功能范围是否超出任务书？

### 7.3 文档一致性

- [ ] PRD 功能描述是否与 UI 设计一致？
- [ ] PRD 字段定义是否与技术方案一致？
- [ ] PRD 数据量是否与模拟数据一致？

### 7.4 模块合理性

- [ ] 每个模块是否有独立价值？
- [ ] 模块划分是否清晰？
- [ ] 是否有"伪模块"（只是操作却设计成模块）？

## 八、关键教训总结

### 8.1 多 Agent 协作

| 教训 | 建议 |
|------|------|
| 不要假设用户需求 | 明确确认每一项设计决策 |
| 服务可能限流 | 预留重试机制 |
| 指令要清晰 | 每个 Agent 只做一件事 |
| 验证输出 | 检查文件是否正确生成 |

### 8.2 文档先行

| 教训 | 建议 |
|------|------|
| 不要跳过评审 | 多轮评审确保质量 |
| 不要自己添加功能 | 用户没说的不要做 |
| 保持文档一致 | 任何修改都要同步相关文档 |
| 评审要全面 | 不仅是格式，还有逻辑和决策 |

### 8.3 数据驱动

| 教训 | 建议 |
|------|------|
| 数据规格要早定 | 避免后期大规模修改 |
| 字段要完整 | 覆盖所有必要场景 |
| 分布要合理 | 模拟真实业务场景 |

## 九、下次项目建议

### 9.1 启动前准备

1. **明确所有关键决策**（技术栈、用户系统、权限、API密钥、部署方式）
2. **建立文档目录结构**
3. **配置多 Agent 协作框架**

### 9.2 执行中

1. **每个阶段有明确的产出和评审**
2. **用户参与关键节点确认**
3. **及时同步文档修改**
4. **记录决策过程到经验文档**

### 9.3 评审重点

1. **功能是否覆盖任务书**
2. **是否存在不需要的功能**
3. **文档之间是否一致**
4. **数据规格是否符合业务场景**
