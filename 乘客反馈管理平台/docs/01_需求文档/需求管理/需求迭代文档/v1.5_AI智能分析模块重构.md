## 1. 概述

### 1.1 背景与目的

当前 AI 智能分析模块存在以下问题：
1. **分析入口不明确** - 缺少统一的"开始分析"入口
2. **无法获取全量数据** - 当前最多只处理 100 条数据，且文本被截断
3. **分析流程不完整** - 缺少系统化的数据清洗 → 摘要 → 问题分析 → 建议的流水线
4. **无分析进度反馈** - 用户无法感知分析进行状态

本迭代重构 AI 智能分析模块，实现完整的多步骤分析流水线。

### 1.2 核心改动

| 改动点 | 当前状态 | 目标状态 |
|--------|----------|----------|
| 分析入口 | 自动加载数据 | 点击"开始分析"按钮触发 |
| 数据筛选 | 无直接关联 FilterBar | 完全基于 FilterBar 筛选条件 |
| 数据规模 | ≤100 条 | 默认 100 条，最多 2000 条 |
| 分析流程 | 单次调用 | 数据清洗 → 摘要 → 问题分析 → 建议 |
| 结果展示 | 两个独立模块 | 三块结果：摘要、问题分类、优化建议 |
| 用户反馈 | 无进度 | 实时进度条展示 |

## 2. 产品逻辑

### 2.1 数据筛选逻辑

数据来源**完全基于**顶部 FilterBar 的筛选条件：

| 筛选结果 | 分析数据量 |
|----------|-----------|
| 筛选结果 > 2000 条 | 取 2000 条进行分析 |
| 筛选结果 ≤ 2000 条 | 取全部筛选结果进行分析 |
| 无筛选条件（点击分析） | 默认取最新 100 条数据 |

**注意**：无需额外的"分析范围"选项，直接使用 FilterBar 的筛选条件。

### 2.2 分析触发流程

```
┌─────────────────────────────────────────────────────────────┐
│  用户在顶部 FilterBar 设置筛选条件                           │
│  用户在 AI 智能分析模块点击「开始分析」按钮                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  前端调用 POST /api/ai/analyze                              │
│  携带 FilterBar 的筛选条件参数                              │
│  后端返回 task_id，状态为 pending                            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  前端展示进度条，每 3 秒轮询 GET /api/ai/analyze/{task_id}  │
│  实时更新分析进度                                            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  分析完成，展示三块结果：                                    │
│  1. AI 摘要 - 整体满意度概况                               │
│  2. 问题分类 - 自动驾驶场景中的产品问题                     │
│  3. 优化建议 - 针对问题的产品迭代建议                       │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 分析流水线

```
Step 1 [0-10%]:  数据获取
                 - 根据 FilterBar 筛选条件查询反馈数据
                 - 无筛选条件时默认取最新 100 条
                 - 筛选结果 > 2000 条时取 2000 条
                 - 筛选结果 ≤ 2000 条时取全部
                 - 文本不截断

Step 2 [10-30%]: 数据清洗
                 - 去除无效反馈（空文本、纯表情等）
                 - 按评分/时间排序
                 - 提取统计数据

Step 3 [30-60%]: 生成 AI 摘要
                 - 整体满意度概况
                 - 正面反馈核心赞扬点
                 - 负面反馈核心痛点

Step 4 [60-85%]: 产品问题分析
                 - 将反馈归类到已有分类
                 - 发现并命名新的问题模式
                 - 统计每个分类的数量、占比、差评率
                 - 按严重程度排序

Step 5 [85-100%]: 生成优化建议
                 - 针对 Top 5 严重问题
                 - 引用用户原声
                 - 给出可落地的优化建议
```

## 3. 功能需求

### 3.1 API 设计

#### POST /api/ai/analyze - 发起分析

**请求参数**：
```typescript
{
  // FilterBar 筛选参数
  start_date?: string,  // YYYY-MM-DD
  end_date?: string,    // YYYY-MM-DD
  city?: string,
  rating_min?: number,
  rating_max?: number,
  status?: string[],
  feedback_type?: string[],
  keyword?: string,
}
```

**响应**：
```typescript
{
  code: 0,
  data: {
    task_id: "a1b2c3d4",
    status: "pending",
  }
}
```

#### GET /api/ai/analyze/{task_id} - 查询分析结果

**响应**：
```typescript
{
  code: 0,
  data: {
    task_id: "a1b2c3d4",
    status: "pending" | "processing" | "completed" | "failed",
    progress: 60,  // 0-100
    summary: "整体满意度良好，好评率75%..." | null,
    problems: [...] | null,
    suggestions: [...] | null,
    analyzed_count: 1500,
    error: "错误信息" | null,
  }
}
```

### 3.2 数据模型

无需新增数据库表。使用内存存储分析任务（server 重启则任务丢失）。

```python
class AnalysisTask:
    task_id: str
    status: str  # pending | processing | completed | failed
    progress: int  # 0-100
    filters: dict  # 原始筛选参数
    created_at: datetime

    # 结果
    summary: str | None
    problems: list | None
    suggestions: list | None
    analyzed_count: int
    error: str | None
```

### 3.3 后端数据查询增强

新增 `FeedbackService.for_analysis()` 方法：

```python
async def for_analysis(
    self,
    # FilterBar 筛选参数
    start_date: Optional[str] = None,
    end_date: Optional[str] = None,
    city: Optional[str] = None,
    rating_min: Optional[int] = None,
    rating_max: Optional[int] = None,
    status: Optional[str] = None,
    keyword: Optional[str] = None,
    feedback_type: Optional[str] = None,
    # 数据量限制
    max_count: int = 2000,  # 无筛选时默认 100，有筛选时最多 2000
) -> Tuple[List[Dict], int]:
    """
    获取用于 AI 分析的完整反馈数据。

    Returns:
        Tuple of (feedbacks list, total estimated count)
    """
```

**关键特性**：
- 返回完整 `feedback_text`（不截断）
- 无筛选条件时：默认取最新 100 条
- 有筛选条件时：最多取 2000 条
- 按 `trip_time` 倒序

### 3.4 Prompt 设计

#### Prompt 1: 生成摘要 (summarize)

```
你是一个专业的Robotaxi乘客反馈分析师。请分析以下反馈数据，生成简洁的摘要报告。

## 数据统计
- 总反馈数：{total_count} 条
- 平均评分：{avg_rating} 分（满分5分）
- 好评率：{positive_rate}%（评分4-5分）
- 差评率：{negative_rate}%（评分1-2分）

## 反馈内容
{feedback_texts}

## 要求
1. 用2-3句话总结整体用户满意度状况
2. 指出用户最常提到的正面体验（不超过2点）
3. 指出用户最突出的不满问题（不超过2点）
4. 整体字数控制在150字以内

## 输出格式
直接输出摘要文本，不需要JSON格式。
```

#### Prompt 2: 产品问题分析 (analyze_problems)

```
你是一个专业的自动驾驶产品分析师。请从以下乘客反馈中识别和分类产品问题。

## 已有分类参考
- 行驶体验：与车辆行驶、加速、刹车、变道等相关
- 车内环境：车内温度、噪音、清洁度、座椅舒适度等
- 接驾体验：等待时间、司机响应、上下车便捷性等
- 路线规划：导航准确性、路线选择、拥堵情况等
- 安全感受：安全带提醒、紧急情况处理、风险预警等
- 服务态度：客服响应、问题解决效率、沟通体验等
- 其他：不属于以上分类的问题

## 反馈数据（按评分排序，低分优先）
{feedback_data}

## 分析要求
1. 将每条反馈归类到上述已有分类（可多选）
2. 识别是否有新的问题模式未覆盖已有分类，如有请命名新分类
3. 统计每个分类的：反馈数量、占比、差评率
4. 按严重程度排序（严重程度 = 差评数量 × 差评率）

## 输出格式（JSON）
{
  "categories": [
    {
      "name": "行驶体验",
      "is_existing": true,
      "count": 45,
      "percentage": 0.15,
      "negative_rate": 0.42,
      "common_issues": ["变道过于频繁", "刹车过于急躁"],
      "user_quotes": ["变道太频繁了，坐着不舒服", "刹车太急"]
    },
    {
      "name": "新发现问题：导航播报",
      "is_existing": false,
      "count": 12,
      "percentage": 0.04,
      "negative_rate": 0.67,
      "description": "导航语音播报过早或过晚，导致错过路口",
      "user_quotes": ["导航说左转但已经过了", "播报太晚来不及变道"]
    }
  ],
  "top_problems": [
    {"category": "行驶体验", "severity_score": 18.9, "problem": "变道过于频繁"},
    ...
  ]
}
```

#### Prompt 3: 优化建议 (generate_suggestions)

```
你是一个自动驾驶产品专家。请基于以下问题分析，生成可落地的产品优化建议。

## 问题分类详情
{problem_categories}

## 差评原声摘录
{negative_feedbacks}

## 要求
1. 针对严重程度最高的前5个问题分类
2. 每个问题给出1-2条具体可落地的优化建议
3. 建议需结合技术实现可能性和用户体验提升
4. 每条建议必须引用1条用户原声作为依据
5. 按严重程度从高到低排序

## 输出格式（JSON数组）
[
  {
    "problem_category": "行驶体验",
    "specific_problem": "变道过于频繁",
    "severity": "high",
    "evidence": {
      "count": 38,
      "negative_rate": 0.71,
      "user_voice": "每次坐车都频繁变道，坐得头晕"
    },
    "suggestions": [
      "优化变道策略，在判断需要变道时优先考虑车道保持而非立即变道",
      "增加变道前的提醒和缓冲时间，让乘客有心理准备"
    ],
    "expected_impact": "降低变道频率X%，提升乘客舒适度评分"
  }
]
```

## 4. 前端设计

### 4.1 组件结构

```
AiAnalysis (页面组件)
├── 分析控制栏
│   ├── 数据预览：基于当前筛选条件，预估 X 条数据
│   └── [开始分析] 按钮
├── 分析进度条（分析中显示）
│   └── 进度百分比 + 当前步骤描述
└── 分析结果区（分析完成显示）
    ├── AI摘要卡片
    ├── 问题分类卡片（DistributionChart）
    └── 产品优化建议卡片
```

### 4.2 交互状态

| 状态 | UI 表现 |
|------|---------|
| 初始 | 显示数据预览 + "开始分析"按钮 |
| 分析中 | 进度条 + 实时进度 + "取消"按钮 |
| 已完成 | 三块结果卡片 + "重新分析"按钮 |
| 失败 | 错误提示 + "重试"按钮 |

```
前端                                        后端
  │                                           │
  │  POST /api/ai/analyze                     │
  │  { filters from FilterBar }              │
  │ ─────────────────────────────────────────► │
  │                                           │
  │  { task_id: "a1b2c3d4" }                  │
  │ ◄───────────────────────────────────────── │
  │                                           │
  │  GET /api/ai/analyze/a1b2c3d4             │
  │ ─────────────────────────────────────────► │
  │  { status: "processing", progress: 30 }   │
  │ ◄───────────────────────────────────────── │
  │       ...轮询直到 completed...             │
  │                                           │
  │  GET /api/ai/analyze/a1b2c3d4             │
  │ ─────────────────────────────────────────► │
  │  { status: "completed", progress: 100,    │
  │    summary: "...", problems: [...],        │
  │    suggestions: [...] }                    │
  │ ◄───────────────────────────────────────── │
```

### 5.2 分析流水线执行

```
Task Runner
  │
  ├─► Step 1: get_feedbacks(filters)
  │       progress: 10%
  │       │
  ├─► Step 2: clean_data(feedbacks)
  │       progress: 30%
  │       │
  ├─► Step 3: generate_summary(cleaned_data)
  │       progress: 60%
  │       │
  ├─► Step 4: analyze_problems(cleaned_data)
  │       progress: 85%
  │       │
  ├─► Step 5: generate_suggestions(problems)
  │       progress: 100%
  │       │
  └─► save_results() → status: completed
```

## 6. 验收标准

### 6.1 功能验收

- [ ] 点击"开始分析"按钮后发起分析请求
- [ ] 进度条正确显示分析进度（0-100%）
- [ ] 分析使用顶部 FilterBar 的筛选条件
- [ ] 无筛选时默认分析最新 100 条数据
- [ ] 筛选结果 > 2000 条时只分析 2000 条
- [ ] 筛选结果 ≤ 2000 条时分析全部数据
- [ ] 分析完成后展示三块结果：摘要、问题分类、优化建议
- [ ] 支持"重新分析"和"取消"操作

### 6.2 数据验收

- [ ] 反馈文本完整，不被截断
- [ ] 统计数据（总数、好评率、差评率）准确
- [ ] 问题分类包含已有分类和新发现分类
- [ ] 优化建议包含用户原声引用

### 6.3 错误处理

- [ ] 网络错误时显示错误提示
- [ ] 分析失败时显示失败原因和重试按钮
- [ ] 取消分析后正确清理状态
