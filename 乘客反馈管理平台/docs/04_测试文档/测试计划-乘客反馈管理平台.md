# 测试计划: Robotaxi 乘客反馈管理平台

## 1. 测试范围与目标

### 1.1 测试范围

本测试计划覆盖 Robotaxi 乘客反馈管理平台的全部功能模块，包括：

| 模块 | 功能点 | 测试范围 |
|------|--------|----------|
| 模块一：反馈列表与管理 | 反馈列表展示 | 分页、排序（时间/评分）功能验证 |
| | 多维度筛选 | 评分范围、时间范围、路线、反馈类型筛选 |
| | 反馈详情 | 单条反馈完整信息展示 |
| 模块二：数据统计仪表盘 | 核心指标卡片 | 总反馈数、平均评分、好评率/差评率计算 |
| | 趋势图表 | 反馈数量及评分时间趋势展示 |
| | 分布分析 | 路线、时间段、反馈类型分布 |
| 模块三：AI 智能分析 | 自动分类 | 五类反馈分类准确性验证 |
| | 摘要生成 | 批量反馈智能摘要效果评估 |
| | 产品建议 | 自动生成优化建议质量评估 |

### 1.2 测试目标

- **功能完整性**：验证所有功能模块按需求规格实现，无遗漏
- **功能正确性**：确保各功能输出结果符合预期，业务逻辑正确
- **界面可用性**：验证前端界面交互流畅，用户体验良好
- **接口稳定性**：确保前后端接口调用稳定，数据传输准确
- **AI 准确性**：验证 AI 分类、摘要、建议功能的准确性达到预期标准
- **性能达标**：系统在合理负载下响应时间满足要求
- **缺陷修复**：发现并跟踪缺陷直至关闭

## 2. 测试策略

### 2.1 测试类型

#### 2.1.1 功能测试

| 测试类型 | 描述 | 适用场景 |
|----------|------|----------|
| 冒烟测试 | 核心业务流程快速验证 | 提测后第一时间进行 |
| 单元测试 | 关键业务逻辑函数验证 | 后端 API 开发阶段 |
| 集成测试 | 多模块联合功能验证 | 前后端联调阶段 |
| 回归测试 | 修改后相关功能验证 | 缺陷修复后常规执行 |

#### 2.1.2 界面测试

- **布局测试**：验证页面元素布局符合设计稿
- **交互测试**：验证按钮、输入框、分页等组件交互正常
- **响应式测试**：验证不同分辨率下的页面展示（桌面端、平板端）
- **兼容性测试**：验证主流浏览器（Chrome、Firefox、Safari、Edge）的兼容性

#### 2.1.3 接口测试

| 测试重点 | 测试内容 |
|----------|----------|
| 参数验证 | 必填项、参数类型、参数范围验证 |
| 返回值验证 | 响应状态码、数据结构、数据内容 |
| 异常处理 | 空值、超时、非法输入的处理 |
| 安全性 | SQL 注入、XSS 攻击防护验证 |

#### 2.1.4 AI 功能测试

| AI 功能 | 测试重点 | 评估指标 |
|---------|----------|----------|
| 自动分类 | 分类准确性、分类置信度 | 准确率 ≥ 85% |
| 摘要生成 | 内容完整性、信息覆盖率 | 核心信息覆盖率 ≥ 80% |
| 产品建议 | 建议相关性、可执行性 | 建议采纳率 ≥ 60% |

#### 2.1.5 性能测试

- **响应时间测试**：各页面加载时间、接口响应时间
- **并发测试**：多用户同时操作时的系统稳定性
- **压力测试**：模拟峰值负载下的系统表现

### 2.2 测试环境

#### 2.2.1 测试环境配置

| 环境类型 | 用途 | 配置要求 |
|----------|------|----------|
| 开发环境 | 开发人员联调 | 本地部署 |
| 测试环境 | 测试人员执行测试 | 与生产环境一致 |
| 预发布环境 | 上线前最终验证 | 1:1 镜像生产 |

#### 2.2.2 测试环境架构

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   前端应用   │ ──▶ │   API 网关   │ ──▶ │   后端服务   │
│  (React)    │     │  (FastAPI)  │     │  (Python)   │
└─────────────┘     └─────────────┘     └─────────────┘
                                               │
                    ┌──────────────────────────┼──────────────────────────┐
                    │                          │                          │
              ┌─────▼─────┐            ┌───────▼───────┐          ┌───────▼───────┐
              │   MySQL   │            │  MiniMax API  │          │  Kimi API     │
              │  数据库    │            │  (AI 分类)    │          │  (摘要生成)   │
              └───────────┘            └───────────────┘          └───────────────┘
```

#### 2.2.3 测试数据要求

- **基础数据**：至少 100 条模拟反馈数据，覆盖各评分等级、反馈类型
- **边界数据**：极端评分（1 分、5 分）、最大文本长度反馈
- **异常数据**：空值、特殊字符、超长文本等

## 6. 缺陷管理

### 6.1 缺陷等级定义

| 缺陷等级 | 定义 | 处理要求 | 示例 |
|----------|------|----------|------|
| **P0 - 致命缺陷** | 系统崩溃、核心功能完全不可用、数据丢失 | 需立即修复，24 小时内解决 | 页面无法加载、API 服务宕机、反馈数据丢失 |
| **P1 - 严重缺陷** | 核心功能严重受损，存在严重业务逻辑错误 | 需优先修复，3 天内解决 | 分页功能失效、评分计算错误、筛选结果不正确 |
| **P2 - 一般缺陷** | 功能正常但存在明显问题，或交互体验不佳 | 正常修复，1 周内解决 | 页面样式偏差、tooltip 显示不完整、响应速度慢 |
| **P3 - 轻微缺陷** | 界面细节问题，不影响功能使用 | 计划修复，2 周内解决 | 文字对齐问题、图标不清晰、缺少 loading 状态 |
